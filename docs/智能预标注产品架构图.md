## L0 · 总览：三方职责全景图

> 整个智能预标注系统由三个相互解耦的主体组成，通过两道 HTTP 契约连接。

```
╔════════════════════════════════════════════════════════════════════════════════╗
║                           智能预标注 · 三方职责全景                              ║
╠════════════════════════════════════════════════════════════════════════════════╣
║                                                                                ║
║   ┌──────────────────────┐        ┌──────────────────────┐        ┌─────────┐ ║
║   │                      │        │                      │        │         │ ║
║   │   🟦  标注平台        │        │   ⚙  ML BACKEND      │        │   🔮    │ ║
║   │   (Label Studio /     │        │   (自写 / 复用)       │        │  LLM   │ ║
║   │    自研平台)          │        │                      │        │  /模型  │ ║
║   │                      │        │                      │        │  API   │ ║
║   │   · 前端 UI          │        │   · 协议翻译层        │        │         │ ║
║   │   · 项目/任务管理     │◀──────▶│   · Prompt 工程      │◀──────▶│ · 推理  │ ║
║   │   · 协议调度          │  HTTP  │   · Few-shot 构造    │  HTTP  │ · 约束  │ ║
║   │   · Prediction 存储  │(LS 定义)│   · 调 LLM API      │(厂商定义)│   输出  │ ║
║   │   · 渲染预标注        │        │   · 回写 LS 协议     │        │         │ ║
║   │   · 审校交互          │        │                      │        │         │ ║
║   │                      │        │                      │        │         │ ║
║   └──────────────────────┘        └──────────────────────┘        └─────────┘ ║
║                                                                                ║
║   开发者:  平台厂商                开发者: 算法/数据团队          供应商:      ║
║   代码量:  大型 Web 应用           代码量: 几百行 Python         OpenAI/自建   ║
║   变更频率:低                     变更频率:中                    变更频率:低   ║
║                                                                                ║
╠════════════════════════════════════════════════════════════════════════════════╣
║                                                                                ║

║                      · JSON Schema (from_name/to_name/type/value)             ║
║                                                                                ║
║  边界 2:  ⚙ ↔ 🔮     LLM 厂商定义的 API                                       ║
║                      · OpenAI Chat Completion                                 ║

║                      · HuggingFace Inference API                              ║
║                                                                                ║
╚════════════════════════════════════════════════════════════════════════════════╝
```

**核心洞察**：

| 维度 | 🟦 平台 | ⚙ ML Backend | 🔮 LLM/模型 |
|---|---|---|---|
| 关注点 | 怎么把任务送进来、把预测显出去 | 怎么把任务翻译成模型能懂的样子，再翻译回来 | 给定输入，按 Schema 输出 |
| 层次 | 交互层 | 编排层 | 算力层 |
| 解耦价值 | 改 UI 不影响模型 | 改模型不影响平台 | 换 LLM 不影响平台 |

## L1 · 分层架构：四层能力堆栈

> 把预标注系统从用户界面一路画到底层推理，共 4 层。

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 🎯 Layer 4: 标注前台 (Labeling UI)                                       │
│ ───────────────────────────────────────────────────────────────────────│
│   · 画布渲染器（按 type 分派：choices/rectangle/brush/textarea…）        │
│   · Smart Tools（智能画笔、智能矩形，调用交互式预标注）                    │
│   · 预标注呈现（虚线/半透明 + 置信度徽章）                                │
│   · 审校交互（接受/拒绝/Y/N 快捷键/一键接受高置信度）                     │
│   · 标注员认领 + 进度追踪 + 质检流程                                      │
└─────────────────────────────────────────────────────────────────────────┘
                                   ▲
                                   │
┌─────────────────────────────────────────────────────────────────────────┐
│ ⚙ Layer 3: 编排层 (ML Orchestration)                                    │
│ ───────────────────────────────────────────────────────────────────────│
│   · 模型管理（注册/连接/健康检测/默认模型）                               │
│   · 任务 ↔ 模型绑定（按模态过滤，标签映射）                              │
│   · 批量预标注调度（任务下发时自动触发）                                  │
│   · 预标注进度追踪（TaskPreannotationConfig）                           │
│   · Webhook 事件派发（ANNOTATION_CREATED / TASK_CREATED…）              │
│   · 主动学习策略（不确定性/多样性/Query-by-Committee）                   │
└─────────────────────────────────────────────────────────────────────────┘
                                   ▲
                                   │
┌─────────────────────────────────────────────────────────────────────────┐
│ 🔌 Layer 2: 协议层 (ML Backend Protocol)                                 │
│ ───────────────────────────────────────────────────────────────────────│
│   · 4 个 HTTP 端点契约 (/setup /predict /train /webhook)                │
│   · 统一 JSON Schema (from_name/to_name/type/value)                     │
│   · Label Config XML（双向声明契约）                                     │
│   · Context 协议（交互式输入）                                           │
│   · extra_params 扩展点                                                 │
│   · 反向回调能力 (get_local_path, /api/tasks)                          │
└─────────────────────────────────────────────────────────────────────────┘
                                   ▲
                                   │
┌─────────────────────────────────────────────────────────────────────────┐
│ 🧠 Layer 1: 模型层 (Model Execution)                                     │
│ ───────────────────────────────────────────────────────────────────────│
│   ┌───────────┐ ┌──────────┐ ┌──────────┐ ┌────────────┐                │
│   │ 封闭集     │ │ 开放集    │ │ 类别无关  │ │ 少样本      │                │
│   │ BERT/YOLO │ │ LLM/CLIP │ │ SAM      │ │ Prototype  │                │
│   └───────────┘ └──────────┘ └──────────┘ └────────────┘                │
│         ↓             ↓             ↓              ↓                    │
│   ┌───────────────────────────────────────────────────────┐             │
│   │      Adapter 层（把各模型原生输出翻译成协议格式）         │             │
│   └───────────────────────────────────────────────────────┘             │
│                            ↓                                            │
│   ┌───────────────────────────────────────────────────────┐             │
│   │  底层算力（GPU / LLM API / 推理框架：vLLM, ONNX, TGI）   │             │
│   └───────────────────────────────────────────────────────┘             │
└─────────────────────────────────────────────────────────────────────────┘
```

**解耦价值**：

- 每一层只依赖下一层的**接口**，不依赖其实现
- 更换底层模型（Layer 1）不影响协议（Layer 2）
- 改协议扩展（Layer 2）只需前端渲染器扩展一种 type
- 编排策略（Layer 3）可独立演化，不触发模型重训

## L2 · 协议契约：ML Backend 接口全景

> 协议是整个系统的"插座"——这张图画清楚了插座的规格。

```
╔═══════════════════════════════════════════════════════════════════════════╗
║                         ML Backend 协议契约总览                             ║
╚═══════════════════════════════════════════════════════════════════════════╝

  🟦 标注平台  ────────── HTTP/JSON ──────────▶  ⚙ ML Backend


  ┌─────────────────┐   POST /setup                ┌────────────────────┐
  │                 │ ─────────────────────────────▶│                    │
  │                 │   { project, schema,         │  ◉ 缓存 label_config │
  │  项目首次绑定    │     hostname, access_token,  │  ◉ 读 extra_params   │
  │                 │     extra_params }           │  ◉ 加载模型/权重      │
  │                 │ ◀─────────────────────────── │  ◉ 回报版本           │
  │                 │   { model_version }          │                    │
  └─────────────────┘                              └────────────────────┘

  ┌─────────────────┐   POST /predict              ┌────────────────────┐
  │                 │ ─────────────────────────────▶│                    │
  │                 │   { tasks[],                 │  ◉ 解析 label_config │
  │  Retrieve       │     project,                 │  ◉ 拉原始文件        │
  │  Predictions    │     label_config,            │  ◉ 构造 Prompt/输入  │
  │  （批量/交互）   │     params: { context } }    │  ◉ 推理              │
  │                 │ ◀─────────────────────────── │  ◉ 适配协议格式      │
  │                 │   { results: [{              │                    │
  │                 │      model_version,          │                    │
  │                 │      score,                  │                    │
  │                 │      result: [{              │                    │
  │                 │        from_name,            │                    │
  │                 │        to_name,              │                    │
  │                 │        type,                 │                    │
  │                 │        value: {...}          │                    │
  │                 │      }]                      │                    │
  │                 │    }] }                      │                    │
  └─────────────────┘                              └────────────────────┘

  ┌─────────────────┐   POST /train                ┌────────────────────┐
  │                 │ ─────────────────────────────▶│                    │
  │                 │   { event,                   │  ◉ 收集 annotations │
  │  标注累积触发    │     annotations[],           │  ◉ 异步训练          │
  │  或 Webhook      │     project,                 │  ◉ 更新权重/Prompt  │
  │                 │     label_config }           │  ◉ bump version     │
  │                 │ ◀─────────────────────────── │                    │
  │                 │   202 Accepted               │                    │
  └─────────────────┘                              └────────────────────┘

  ┌─────────────────┐   POST /webhook              ┌────────────────────┐
  │                 │ ─────────────────────────────▶│                    │
  │  事件总线        │   { event: ANNOTATION_*,     │  ◉ 订阅感兴趣事件    │
  │                 │     data: {...} }            │  ◉ 触发后续动作      │
  └─────────────────┘                              └────────────────────┘

  ┌─────────────────┐   GET /                      ┌────────────────────┐
  │                 │ ─────────────────────────────▶│                    │
  │  健康检查        │                              │  ◉ 返回状态与版本    │
  │                 │ ◀─────────────────────────── │                    │
  │                 │   { status: "UP",            │                    │
  │                 │     model_class: "..." }     │                    │
  └─────────────────┘                              └────────────────────┘


  ⚙ ML Backend  ────── 反向回调 (可选) ──────▶  🟦 标注平台

  ┌──────────────┐   GET /api/tasks/{id}           ┌──────────────────┐
  │ 模型拉数据    │ ──────────────────────────────▶ │                  │
  │              │   + Token                       │ 平台提供数据访问   │
  │              │ ◀────────────────────────────── │                  │
  │              │   二进制文件 / JSON              │                  │
  └──────────────┘                                 └──────────────────┘
```

**协议字段职责分解**：

| 字段 | 由谁决定 | 作用 |
|---|---|---|
| `from_name` | XML `<Choices name="xxx">` 定义 | 前端找到工具实例的键 |
| `to_name` | XML 工具的 `toName` 属性 | 指向要标注的数据对象 |
| `type` | 标注工具类型映射（choices/rectanglelabels/brushlabels…） | 前端分派渲染器 |
| `value` | 由 type 决定结构 | 真正的标注内容 |
| `score` | 模型自报 | 置信度，用于筛选/排序 |
| `model_version` | 模型自报 | 支持 A/B 对比 |

**四档模型对 `context` 字段的不同消费方式**：

| 模型档位 | 批量预标注 | 交互式预标注 | context 作用 |
|---|:---:|:---:|---|
| 🔒 封闭集 | ✅ 主力场景 | ❌ 一般不支持 | 忽略 |
| 🌐 开放集 | ✅ | ✅ | 转为"用户追问"继续对话 |
| ✂️ 类别无关 | ❌ 不支持（无类别概念） | ✅ 核心场景 | 读 `context.result` 作为点/框提示 |
| 🎯 少样本 | ✅（前提：原型集非空） | ✅ | 扩充支撑集 |

## L4 · 标签协议：三段式翻译链路

> 聚焦一个最容易混淆的问题：**标签从 XML 出发，最终如何变成模型输出？**

```
┌────────────────────────────────────────────────────────────────────────────┐
│         标签的三段式翻译：从 UI 到协议，再到模型执行                          │
└────────────────────────────────────────────────────────────────────────────┘

  【第一段】UI 声明 → label_config XML
  ─────────────────────────────────────

  标注工具编辑器 UI                       label_config (XML)
  ┌─────────────────────────┐            ┌────────────────────────────┐
  │ 标注对象: [文本]          │            │ <View>                     │
  │                         │            │   <Text name="txt"          │
  │ 标注方法: [单选]          │    ──▶    │         value="$text"/>    │
  │                         │            │   <Choices name="sentiment" │
  │ 标签:                   │            │           toName="txt"      │
  │  [正面] [负面] [中性]    │            │           choice="single"> │
  │                         │            │     <Choice value="正面"/>  │
  │                         │            │     <Choice value="负面"/>  │
  │                         │            │     <Choice value="中性"/>  │
  │                         │            │   </Choices>                │
  └─────────────────────────┘            │ </View>                     │
                                         └────────────────────────────┘

  【第二段】label_config → 模型侧内部表示（按档位不同而异）
  ──────────────────────────────────────────────────────

  ┌────────────────────────────────────┐
  │ 🔒 封闭集：映射表                   │
  │ {                                  │
  │   "POS": "正面",                   │
  │   "NEG": "负面",                   │
  │   "NEU": "中性"                    │
  │ }                                  │
  └────────────────────────────────────┘

  ┌────────────────────────────────────┐
  │ 🌐 开放集：Prompt 模板              │
  │ """                                │
  │ 分类以下文本，选项:                  │
  │ - 正面                              │
  │ - 负面                              │
  │ - 中性                              │
  │                                    │
  │ 文本: {input}                      │
  │ """                                │
  │                                    │
  │ + Function Calling Schema:         │
  │ { "label": enum[正面,负面,中性] }   │
  └────────────────────────────────────┘

  ┌────────────────────────────────────┐
  │ ✂️ 类别无关：只记录位置             │
  │ from_name="sentiment"              │
  │ to_name="txt"                      │
  │ (忽略具体标签值)                     │
  └────────────────────────────────────┘

  ┌────────────────────────────────────┐
  │ 🎯 少样本：原型集                    │
  │ {                                  │
  │   "正面": [vec₁, vec₂, vec₃...],   │
  │   "负面": [vec₄, vec₅...],         │
  │   "中性": []  (冷启动为空)          │
  │ }                                  │
  └────────────────────────────────────┘


  【第三段】模型输出 → 协议 JSON → UI 渲染
  ────────────────────────────────────

  模型原生输出               协议 JSON                     UI 渲染结果
  ┌──────────────┐          ┌──────────────────────┐     ┌───────────────────┐
  │ 封闭集:       │          │ {                    │     │                   │
  │ {            │          │   from_name:         │     │  ● 正面 (预测 82%)│
  │   logits:    │          │     "sentiment",     │     │  ○ 负面           │
  │   [0.82,     │          │   to_name: "txt",    │ ──▶ │  ○ 中性           │
  │    0.12,     │ ───▶     │   type: "choices",   │     │                   │
  │    0.06]     │          │   value: {           │     │  [接受] [拒绝]    │
  │ }            │          │     choices: ["正面"]│     │                   │
  └──────────────┘          │   },                 │     └───────────────────┘
                            │   score: 0.82        │
                            │ }                    │
                            └──────────────────────┘
```

**三段式翻译的价值**：

| 段 | 职责 | 稳定性 | 修改影响 |
|---|---|---|---|
| 第一段（UI → XML） | 标注工具定义 | 修改频繁 | 只影响自己的任务 |
| 第二段（XML → 模型内部） | Adapter 适配 | 每种模型一次 | 不影响其他档位模型 |
| 第三段（模型输出 → 协议 → UI） | 协议编解码 | 几乎不变 | 影响全平台（因此必须稳定） |

## 附录 A：核心架构原则（一页总结）

```
┌────────────────────────────────────────────────────────────────────────┐
│                   智能预标注架构的 5 条黄金原则                          │
├────────────────────────────────────────────────────────────────────────┤
│                                                                        │
│ 1. 协议大于 SDK                                                         │
│    HTTP + JSON Schema 是合同，SDK 是参考实现。任何语言都能接入。          │
│                                                                        │
│ 2. 类型擦除                                                             │
│    用 type 字段做鉴别器，让前端不关心模型是什么，只关心怎么渲染。         │
│                                                                        │
│ 3. 配置驱动                                                             │
│    label_config (XML) 是声明式的，下发给模型后模型按此适配自己的输出。    │
│                                                                        │
│ 4. 双轨数据                                                             │
│    Prediction 与 Annotation 并存，永不覆盖。留下事后复盘的全部证据。     │
│                                                                        │
│ 5. 无状态 + 反向回调                                                    │
│    协议本身不传大数据，模型按需回拉。避免 HTTP 体积膨胀。                 │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

## 附录 B：关键接口速查

```
🟦 Label Studio 对 ⚙ ML Backend 的调用:
  · POST /setup       项目首次绑定时下发 label_config
  · POST /predict     请求推理，batch 或 interactive 两种模式
  · POST /train       训练端点（可选）
  · POST /webhook     事件推送（ANNOTATION_CREATED 等）
  · GET  /            健康检查

⚙ ML Backend 对 🟦 Label Studio 的调用:
  · GET  /api/tasks/{id}        拉任务详情
  · GET  /api/annotations       拉历史标注（作 few-shot）
  · GET  /data/upload/...       拉上传的原始文件
  · POST /api/predictions       主动写入预测

⚙ ML Backend 对 🔮 LLM API 的调用:
  · OpenAI Chat Completion + Function Calling
  · OpenAI Structured Outputs (Pydantic)
  · HuggingFace Transformers Pipeline
  · 本地 vLLM/SGLang/TGI 推理框架

协议核心字段:
  · from_name / to_name  — 绑定工具实例
  · type                 — 鉴别结果形状
  · value                — 实际标注内容
  · score                — 置信度
  · model_version        — 模型版本
  · context              — 交互式输入
  · extra_params         — 模型自定义配置
```
