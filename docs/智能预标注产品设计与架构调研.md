## 一、数据预标注功能的产品设计

### 1. 价值主张与目标拆解

#### 1.1 核心价值
预标注（Pre-annotation / Auto-labeling）的本质是**让模型先干脏活累活，人只做审核与修正**，从而把人工标注从"从零生产"升级为"增量审校"。

#### 1.2 目标用户与痛点
| 角色 | 痛点 | 预标注带来的收益 |
|---|---|---|
| 标注员 | 重复劳动、效率瓶颈 | 把"画框/打标签"变成"点确认/微调" |
| 项目经理 | 人力成本高、交付周期长 | 同等人力下吞吐量 2-10x |
| 算法工程师 | 冷启动数据贵，迭代慢 | 形成"标注→训练→预标注→再标注"闭环（Active Learning） |
| 审核员 | 质量抽查效率低 | 基于置信度排序优先审核低质数据 |

#### 1.3 成功指标（北极星 + 辅助）
- 北极星：**单位任务平均人工耗时（sec/task）** 下降 ≥ 40%
- 辅助：预标注接受率（Accept Rate）、模型置信度与真实准确率一致性、任务吞吐量、标注成本/千条

### 2. 功能架构（借鉴 Label Studio 的核心分层）

Label Studio 把 ML 能力抽象成了三层，这是一个非常值得沿用的骨架：

```
┌─────────────────────────────────────────────┐
│  1) 标注前台 (Labeling UI)                   │
│     · 预测结果渲染  · Smart Tools  · 审校交互 │
├─────────────────────────────────────────────┤
│  2) 编排层 (ML Orchestration)                │
│     · 模型管理  · 触发策略  · 置信度路由      │
├─────────────────────────────────────────────┤
│  3) ML Backend (模型服务)                    │
│     · /predict  · /train  · /health          │
└─────────────────────────────────────────────┘
```

### 3. 功能模块详细设计

#### 3.1 模块 1：模型接入与管理

**功能点：**
1. **模型注册**：项目维度配置，一个项目可挂多个模型（如"粗筛模型 + 精标模型"）
2. **模型类型枚举**：
   - 自研模型（HTTP 协议对接，平台只规定输入输出 Schema）
   - 第三方 API（OpenAI / Azure / 自家 LLM 网关）
   - 平台内置模型市场
3. **健康检查**：UI 上提供 "Send Test Request" 按钮
4. **多版本管理**：支持一个模型 N 个版本，切换时不影响历史预测结果的可追溯性

**产品决策建议：**
- MVP 只支持 HTTP 协议的外部模型接入（降低工程成本）
- 内置 3-5 个开箱即用模型（建议优先级：通用图像检测、OCR、文本分类、NER、LLM 指令抽取）

#### 3.2 模块 2：预标注生成模式

##### 2.1 批量预标注（Batch Pre-annotation）

**使用场景**：整批数据导入后，先跑一遍模型，再分配给标注员

**交互设计：**
- 在数据管理页（Data Manager）选中任务 → Actions → **"生成预标注"**
- 或在项目设置中开启 **"新任务自动预标注"** 开关（订阅任务创建事件）
- 任务量大时使用异步任务队列，UI 展示进度条与失败重试

**注意点：**
- 单次 HTTP 请求做全量预测容易超时 → **按 task 粒度 POST 到 `/predictions`**，由后端队列调度
- 要提供"删除预标注"能力（误操作、换模型后回滚）

##### 2.2 交互式预标注（Interactive Pre-annotation / Smart Tools）

**使用场景**：标注员边标边让模型辅助，典型如"点一下前景，SAM 自动分割出整个物体"

**交互设计：**
- 标注工具栏复制一份带 ⚡ 图标的"智能版本"
- 标注员操作时（画框、点点、输 prompt），前端把 **context**（当前操作 + 当前区域）发给 `/predict`
- 模型返回建议区域 → 前端渲染为"待确认"态
- 提供两种模式：自动接受建议 / 手动审校

**产品决策建议：**
- 默认**不自动接受**，避免"模型污染标注"

#### 3.3 模块 3：预测结果的呈现与审校

**设计要点：**
1. **区分预测与标注的视觉态**：预测 = 虚线/半透明/带 AI 标识；已确认标注 = 实线/实心
2. **置信度透出**：每条预测附带 confidence score
3. **操作最短路径**：快捷键一键接受（如 `A`）、一键拒绝（`D`）、接受全部（`Shift+A`）
4. **多模型对比**：同一任务可叠加多个模型预测，标注员选择最优的那个作为起点

#### 3.4 模块 4：训练闭环（Active Learning）

**功能设计：**
1. **训练触发方式**：手动 / 自动（Webhook 监听事件达阈值触发）/ 定时（CRON）
2. **训练状态可视化**：训练进度、loss 曲线、上次训练时间、本次训练所用数据量
3. **主动学习（Active Learning）**：模型对未标注任务打分 → 按"不确定性 / 多样性"排序 → 优先分配给标注员

#### 3.5 模块 5：数据与资源访问

- 模型需要拉取原始数据（图片、音频、视频）
- 数据源有 4 类：直连 HTTP、平台上传、本地存储、云存储（S3/GCS/Azure）
- **平台必须向模型服务下发**：`PLATFORM_URL` + `API_KEY`

**产品侧要做的事：**
- 在模型接入 UI 里自动生成并提示环境变量配置
- 为云存储文件提供"预签名 URL"能力
- 权限隔离：模型只能拉取当前项目授权的数据

### 4. MVP 范围与迭代路线

| 阶段 | 范围 | 目标 |
|---|---|---|
| **V1 MVP** | 模型注册（HTTP）+ 批量预标注 + 基础审校 UI + 置信度展示 | 验证"预标注可以节省工时"这一核心假设 |
| **V2** | 交互式预标注（Smart Tools）+ 至少支持图像矩形/多边形 + 内置 3 个模型 | 覆盖高价值场景 |
| **V3** | 训练闭环（/train + Webhook）+ 主动学习任务排序 + 多版本模型管理 | 形成数据飞轮 |
| **V4** | LLM 预标注 + 模型评估对比 | 对齐 Label Studio Enterprise 的 "Prompts" 能力 |

### 5. 与 Label Studio 的差异化机会点

1. **模型市场 / 开箱即用**：Label Studio 需要用户 `git clone` + `docker-compose up`，门槛高
2. **主动学习开箱即用**：Label Studio 开源版没有，Enterprise 才有
3. **LLM-as-a-labeler 的配置化**：提供 Prompt 模板、Schema 约束解析、Few-shot 示例管理
4. **质量监控闭环**：预标注接受率 / 一致性 / 漂移告警
5. **标注员视角的"智能建议解释"**：把模型预测的依据展示出来

### 6. 关键风险与应对

| 风险 | 应对 |
|---|---|
| 模型质量差导致标注员被误导（"anchoring effect"） | 默认关闭"自动接受"；提供 A/B 对照组评估 |
| 模型服务稳定性影响标注流程 | 健康检查 + 超时降级 |
| 数据权限泄漏给外部模型 | 预签名 URL + 项目级隔离 + 审计日志 |
| 置信度分数不可信（校准差） | 提供"置信度校准"工具 |

### 7. 核心思想总结

> **预标注不是给模型打工，而是把标注员从"生产者"升级为"审校者"。**
> 设计所有交互都应围绕"如何让人更快更准地接受或拒绝模型建议"展开。

## 二、Label Studio ML 预标注完整流程示例

以一个**"宠物图像目标检测"**场景走完从零到闭环的全流程。

> **场景设定**：宠物 APP 的数据团队，10,000 张宠物照片，需要标注出图中每只动物的边界框和种类（cat / dog / bird）。

### 阶段 0：准备工作（一次性）

1. 启动 Label Studio 主服务
2. 创建项目 "Pet Detection"，配置标注界面（Object Detection with Bounding Boxes）
3. 导入数据：每条记录分配 `task_id`（如 1001、1002、1003）

此时还没有任何标注，也没有任何预测。

### 阶段 1：启动 ML Backend（算法工程师视角）

1. 克隆 ML Backend 仓库，选用 `mmdetection-3` 示例
2. 配置 `docker-compose.yml` 里的环境变量（最关键的是 `LABEL_STUDIO_URL` 和 `LABEL_STUDIO_API_KEY`）
3. 启动 docker-compose → 模型服务在 `http://localhost:9090` 健康运行

### 阶段 2：把模型接入项目

在项目设置 → Model → "Connect Model"，填 URL 并保存。Label Studio 会：
1. 对该 URL 发一个 `POST /setup` 请求，下发 `project_id` 和 `label_config`
2. 模型服务记住当前项目的标签 schema

### 阶段 3：批量预标注（Batch Pre-annotation）

1. 进入 Data Manager → 勾选 3 条任务 → **Actions → "Retrieve Predictions"**
2. Label Studio 把 3 条任务打包成一次 HTTP 请求发给模型 `/predict` 端点
3. 模型内部：
   - 用 `get_local_path()` 下载图片
   - 对每张图跑目标检测
   - 把结果封装成 Label Studio 协议格式返回
4. Label Studio 把返回的预测存入 `Prediction` 表，**与原任务关联但独立于人工标注**

### 阶段 4：标注员开始审校

1. 标注员打开第一条任务，看到预先画好的边界框（虚线/半透明 = 来自模型的 prediction）
2. 标注员可以确认、拖动微调、或删除重画
3. 提交后，Label Studio 创建新的 `Annotation`（独立于 Prediction）

**双轨数据结构：**
```
Task 1001
 ├── Prediction  (from model, 不变)
 │     └── cat [x:23.5, y:18.2, w:34.1, h:52.0]
 └── Annotation  (from annotator, 最终产物)
       └── cat [x:24.0, y:18.5, w:33.5, h:51.8]   ← 标注员微调过
```

这个双轨设计很重要：**预测永远保留，方便事后对比"模型 vs 人工"的差异**。

### 阶段 5：自动预标注 + 主动学习

开启自动化：
1. 项目设置开启 `Use predictions to prelabel tasks` 和 `新任务自动生成预标注`
2. 再导入新数据 → Label Studio 自动对每条任务调用 `/predict`
3. 标注员打开任意一张，看到的已经是带预标注的状态
4. 对 `score < 0.5` 的任务打标签"low_confidence"，优先分配给高级标注员（主动学习思想）

### 阶段 6：训练闭环（形成数据飞轮）

1. 触发训练（手动点 "Start Training" 或 API）
2. Label Studio 把所有已标注任务打包后 POST 给模型 `/train` 端点
3. 模型内部执行训练并保存新权重
4. 更新 `model_version`，下次调用 `/predict` 就是用新权重

形成闭环：
```
人工标注 100 条
   ↓
/train 训练得到 v2 模型
   ↓
/predict 给新 500 条打预标注（质量比 v1 更好）
   ↓
人工审校（工作量比 v1 阶段少 30%）
   ↓
/train 训练得到 v3 模型
   ↓
... 越训越好，越标越快 ...
```

### 阶段 7：交互式预标注（Smart Tools）

1. 连接模型时勾选 `Interactive preannotations`
2. 标注界面使用 `smart="true"` 标签
3. 标注员点击操作时前端实时发请求给模型
4. 模型返回建议 → 标注员接受或拒绝

**Batch 模式是任务级别的预标注，Interactive 模式是操作级别的实时辅助。两者并不冲突，可以同时开启。**

### 关键概念速查表

| 概念 | 角色 | 本质 |
|---|---|---|
| **Task** | Label Studio 里的一条待标数据 | JSON 记录 + 唯一 ID |
| **Annotation** | 人工标注结果 | 存在主库，是最终产物 |
| **Prediction** | 模型预测结果 | 存在主库，与 Annotation 并存 |
| **ML Backend** | 一个独立 HTTP 服务 | 实现 `/predict`、`/train`、`/setup`、`/webhook` 四个端点 |
| **Batch 模式** | 对任务列表整批预测 | 触发点：DataManager 的 Action 或新任务导入事件 |
| **Interactive 模式** | 标注员操作时实时预测 | 触发点：Smart 工具的每次点击/绘制 |
| **context** | Interactive 模式下的实时输入 | "我点了这个点"、"我画了这个框" |
| **model_version** | 模型标识 | 同一任务可以有多个 prediction 属于不同版本 |
| **from_name / to_name** | 预测结果里的关键字段 | 必须与标签配置 XML 里的 name 属性严格对应 |

### 最容易踩的 3 个坑

1. **`LABEL_STUDIO_URL` 不能写 `localhost`**：Docker 容器里 `localhost` 指向自己，拉不到主服务数据。必须用主机真实 IP 或 `host.docker.internal`
2. **`from_name/to_name` 不匹配**：模型返回的结果里这两个字段必须和 XML 标签完全一致，否则 UI 静默不显示
3. **大批量预测超时**：不要在一个 HTTP 里预测 10000 条。应按 task 粒度调 `/api/predictions`，由队列异步消费

## 三、在现有平台上设计智能预标注功能

### 业务流程设计

基于平台现状设计的智能预标注完整链路：

```
[1] 模型管理菜单连接模型
        ↓
[2] 新建标注任务时勾选「批量预标注」「交互式预标注」+ 选择模型
    （按模态过滤、默认带常用模型）
        ↓
[3] 任务发布 → 后端按批次自动触发预标注
        ↓
[4] 标注员从「任务大厅」实时看到「预标注 X%」「已预标注 100%」徽章
    → 认领进入「我的任务」
        ↓
[5] 「我的任务」展示每个批次的 预标注数 / 总数 + 模型名 + 实时进度条
        ↓
[6] 点「执行」进入工作台 → 顶部预标注状态条 + 当前样本预标注卡
    （接受/拒绝/置信度/低阈值告警）
        ↓
[7] 一键接受所有高置信度 / Y、N 快捷键 / 接受后自动进入下一题
```

### 数据模型设计

#### MLModel（模型定义）
```
· id / name / description / modality（模态）
· taskTypes（任务类型标签）
· backendUrl / authType / authValue
· supportsBatch / supportsInteractive / supportsTraining
· avgInferenceMs / defaultConfidence
· health / isDefault
· version
```

#### TaskPreannotationConfig（任务预标注配置）
```
· taskId / batchEnabled / interactiveEnabled
· modelId / modelName / modality
· confidenceThreshold / autoAccept
· total / preannotated / failed
· status（未开启/排队中/执行中/已完成/部分失败）
· dispatchedAt / estimatedFinishAt / finishedAt
```

### 关键设计决策

1. **数据双轨**：预测 与 人工标注 数据结构分离，预测保留完整以便后续做模型质量评估
2. **模态强约束**：模型只能选与任务类型兼容的，跨模态模型可跨场景复用
3. **不可用降级**：模型未接入或不支持批量/交互时自动禁用对应开关
4. **不阻塞**：预标注异步执行，标注员任何时候都可作业
5. **接受后自动落库**：接受预标注 → 写入正式标注记录，组别标记为「AI 预标注」、来源为模型名，进入标注列表参与质检流程

### 页面能力分布

| 页面 | 新增能力 |
|---|---|
| **模型管理页** | 模型卡片网格 / 连接弹窗 / 能力声明（批量/交互/训练）/ 健康检测 / 设为默认 / 按模态筛选 |
| **任务创建向导** | 任务配置步骤新增"智能预标注"区块：模型选择、置信度阈值、自动接受、预估耗时；发布时按批次触发预标注 |
| **任务大厅** | 批次列表新增"预标注中 X%"/"已预标注 100%"/"交互"徽章；批次详情显示预标注进度条 |
| **我的任务** | 表格新增"预标注"列，实时显示已预标注/总数+模型名+交互标记 |
| **标注工作台** | 顶部工具栏增加模型标识；顶部状态条（进度+统计+一键接受按钮）；画布顶部预标注建议卡；底部样本导航带预标注状态点；Y/N 快捷键；已接受提示 |

## 四、标注工具与模型的适配问题

### 问题本质：两套独立演化的词汇表

#### 模型侧的世界
模型说的是它自己的"方言"：
- 分类模型：`{label: "positive", confidence: 0.87}`
- 检测模型：`[{bbox, class, confidence}]`
- 分割模型：`{mask, class}`
- NER 模型：`[{text, start, end, type}]`
- ASR 模型：`{transcription}`
- LLM：`{content}`

#### 标注工具侧的世界
标注工具说的是它配置时定义的"方言"：
- 单选标签：`["正面", "负面", "中性"]`
- 矩形框：`[{x%, y%, w%, h%, labels}]`
- 多边形：`[{points, labels}]`
- 笔刷：`{rle, labels}`
- 文本 span：`[{start, end, labels}]`
- 转写文本框：`{text}`

**两套词汇表永远不可能天然对齐**——这就是 Adapter（适配器）层要解决的问题。

### Label Studio 的解法：统一预测协议

Label Studio 的聪明之处在于——**它不让模型去适配工具，而是让所有模型输出同一个 JSON 协议**，由前端根据 `type` 字段渲染。

关键是 **`type` 枚举值**，每种标注工具对应一个固定 `type`：

| type | 匹配的标注工具 |
|---|---|
| `choices` | 单选/多选 |
| `labels` | 文本 span 标签 |
| `rectanglelabels` | 矩形框 |
| `polygonlabels` | 多边形 |
| `brushlabels` | 画笔/像素 |
| `keypointlabels` | 关键点 |
| `textarea` | 转写/自由文本 |
| `rating` | 评分 |
| `pairwise` | 两两比较 |
| `taxonomy` | 树形选择 |

**`from_name` 和 `to_name` 是工具实例的"绑定键"**，必须与标注工具配置里的 XML `name` 属性严格对齐。

前端拿到预测后，按 `from_name + to_name + type` 的元组找到对应的工具实例，把 `value` 注入进去 → 自动渲染为虚线预标注框。**工具和模型之间没有直接耦合，只通过这份协议握手。**

### 三个必须解决的转换点

**① 坐标系转换**
- 模型输出绝对像素 → 前端需要百分比
- 视频坐标需要加 `frame` 字段
- 多边形点序需要符合右手/左手约定

**② 标签映射 (Label Mapping)**
模型训练时的类别（如 COCO 的 80 类）和任务定义的标签（如"车辆/行人"）大概率不一样：

```
模型原生:   [person, bicycle, car, motorcycle, bus, truck, airplane, ...]
                ↓ 映射表
任务定义:   [行人, 车辆, 大型车辆]
```

**这个映射表应该是任务创建时配置的**，不是模型固化的。

**③ 置信度/元数据透传**
模型的 `score` 要保留到 `prediction.score`，用于后续的置信度筛选、主动学习排序。

### 当模型和工具"部分匹配"时怎么办

**情况 A：模型更强大（输出超出工具所需）**
> 处理：通过标签映射表过滤多余类别

**情况 B：模型更弱（输出不足）**
> 处理：降级使用 —— 比如矩形框作为多边形的"4 点起点"，标注员再拖拽调整

**情况 C：结构不一致（形状根本对不上）**
> 处理：**不允许选用**。在任务创建时直接置灰

**情况 D：多工具混合任务**
> 处理：一个任务**可挂多个模型**，每个模型负责一种 `predictionType`

### 产品设计建议

#### 给每个标注方法声明"可消费的预测类型"
这是**工具侧的能力声明**，为每种标注方法加一个 `consumes` 元数据。

#### 给每个模型声明"输出能力"
细粒度的输出类型声明：原生标签空间、坐标格式、是否支持置信度等。

#### 任务创建时做"模型—工具兼容性校验"
按两个维度过滤：
1. 模态必须匹配
2. 模型输出类型必须与工具消费类型有交集

## 五、模型标签范围与四档分类

### 按"标签灵活度"给模型分四档

#### 第一档：封闭集（Closed-Set）—— 标签训练时写死

模型出厂前就定好了 N 个类别，推理时只能在这 N 个里选。

**代表模型**：
| 模型 | 固定标签数 | 举例 |
|---|---|---|
| ImageNet 分类器 | 1,000 类 | golden retriever, volcano... |
| YOLOv8 (COCO 预训练) | 80 类 | person, car, dog... |
| BERT 情感三分类 | 3 类 | positive, negative, neutral |
| 传统 NER (CoNLL-2003) | 4 类 | PER, LOC, ORG, MISC |

**特征**：
- ✅ 输出稳定、精度高、推理快
- ❌ 想加新类必须重新训练
- ❌ 标签名称是"训练时定的英文"

#### 第二档：可配置集（Configurable via Mapping）—— 标签可映射/过滤

本质是封闭集，但通过"标签映射表"让它能服务不同任务。

**这档不是模型自己灵活，而是平台通过映射表把封闭集"裁剪"成任务需要的标签集。**

#### 第三档：开放集（Open-Vocabulary）—— 标签推理时指定

你在调用时**用自然语言或示例告诉模型你要什么类**，模型就能输出对应的预测。

**代表模型**：
| 模型 | 能力 | 怎么指定标签 |
|---|---|---|
| **CLIP / 中文 CLIP** | 图像零样本分类 | 传入文本列表 `["一只猫", "一只狗", "一辆车"]` |
| **Grounding DINO** | 开放词汇目标检测 | 传入 prompt `"cat. dog. traffic cone."` |
| **Segment Anything (SAM)** | 类别无关分割 | 不输出类别，只输出 mask |
| **OWLv2** | 开放词汇检测 | 文本 prompt |
| **GPT-4 / Claude / 国产 LLM** | 文本分类/抽取/打标 | Prompt 工程 + Schema 约束 |
| **Qwen-VL / InternVL** | 多模态理解 | 用 prompt 描述任务 |

**特征**：
- ✅ 一个模型服务无限任务，冷启动无需训练
- ✅ 新增标签零成本（加一行文本即可）
- ❌ 精度弱于同规模封闭集模型
- ❌ 需要**提示词工程**
- ❌ 标签之间容易混淆

#### 第四档（特殊）：少样本/可训练（Few-Shot / Fine-tunable）

介于前两者之间。用少量示例"教会"模型新类别。

- **原型网络**：给每个新类提供 5-20 张样本
- **Fine-tuning**：在基础模型上用 100-1000 条任务数据做轻量微调
- **In-context Learning (LLM)**：Prompt 里带几条示例

### 映射关系：模型档位 ↔ 产品能力要求

| 模型档位 | 标签怎么来 | 平台要给什么 UI/能力 |
|---|---|---|
| **封闭集** | 模型自带 | ① 展示原生标签列表 ② 标签映射表 ③ 过滤/合并能力 |
| **可配置集** | 映射表 | 在封闭集基础上，强化映射 UI（批量、正则、同义合并） |
| **开放集** | 调用时传入 | ① **Prompt 模板编辑器** ② 标签列表直接从任务标签同步 ③ Prompt 版本管理 ④ 置信度校准 |
| **少样本** | 示例样本 | ① **示例管理器** ② 从已标注数据中抽样作为支撑集 ③ 触发重训按钮 |

### 实战选型建议

**场景 A：标签稳定 + 数据量大 + 精度要求高**
> 金融情感、医疗影像分类、车辆检测

**选封闭集模型** + 标签映射。先做小样本冷启动，累积标注数据后走 /train 做 fine-tune。

**场景 B：标签频繁变化 + 冷启动**
> 电商新品分类、内容审核新品类、临时调研项目

**选开放集模型**。无需训练数据，直接通过 Prompt 定义标签。

**场景 C：对象分割 + 类别多变**
> 工业质检、医疗影像

**选 SAM + 分类器组合**。SAM 负责精确分割边界，分类器负责贴标签。

**场景 D：超长尾标签 + 少样本**
> 稀有病识别、小众物种标注

**选少样本模型**。每类提供 5-20 个示例即可上线。

### 核心结论

> **"标签范围是固定的吗？"这个问题本身已经过时了。**
> 更准确的问法是：**"这个模型的标签是怎么定义的？"**
> 
> - 封闭集 → 训练时定
> - 开放集 → 调用时传
> - 类别无关 → 靠下游模型补
> - 少样本 → 靠示例学

**Label Studio 本身不"自带"任何模型**——它是一个连接器。它官方维护的示例模型库**把四档模型全都覆盖了**。

**统计分布**：
- 🔒 封闭集：9 个（60%）—— 仍是主力
- 🌐 开放集：4 个（27%）—— 占比快速提升
- ✂️ 类别无关：1 个（SAM）
- 📏 规则型：1 个

### 三个"明星示例"深度解读

#### 1. `segment_anything_model`（SAM）—— 类别无关模型的标杆
- 能力：给一个点/一个框 → 返回精准的像素级 mask
- 不能：不知道 mask 里的东西叫什么
- 业务价值：把"画一圈轮廓"这个最耗时的操作缩短 10 倍以上

#### 2. `llm_interactive` —— 开放集模型的标杆
- 能力：调用 OpenAI / Azure，用 Prompt 做任意任务
- 标签范围：完全由 Prompt 里的标签列表决定
- 业务价值：冷启动零样本预标注；对新任务无需训练

#### 3. `bert_classifier` —— 封闭集模型的标杆
- 能力：对文本做分类预测
- 标签范围：训练时写死，但支持 /train 端点自我训练
- 业务价值：标注积累到一定量后，按钮一点就能重训，形成数据飞轮

### 行业趋势洞察

```
2022 年之前：以 封闭集 为主（BERT, YOLO, spaCy...）
2023 年：SAM 加入 → 类别无关成为独立品类
2024 年至今：LLM 相关快速增加
```

这反映了一个重要趋势：**预标注正在从"封闭集 + 训练闭环"转向"开放集 + Prompt 工程"**。

## 七、Label Studio"插座"架构揭秘

### 整体架构

```
┌───────────────────────────────────────────────────────────┐
│                    Label Studio 主服务                      │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐     │
│  │  Web UI      │  │  Django API  │  │  Data Manager│     │
│  └──────────────┘  └──────────────┘  └──────────────┘     │
└──────────┬─────────────────────────┬──────────────────────┘
           │                         │
           │  HTTP/JSON              │ HTTP/JSON
           │  (协议)                 │ (协议)
           ▼                         ▼
┌─────────────────────────────────────────────────────────┐
│  ML Backend（独立进程/容器）                              │
│     POST /setup      → 下发 label_config                │
│     POST /predict    → 推理                             │
│     POST /train      → 训练                             │
│     POST /webhook    → 订阅事件                         │
│     GET  /versions   → 模型版本列表                     │
│     GET  /           → 健康检查                         │
└─────────────────────────────────────────────────────────┘
           │
           │ 反向回调（可选）
           │ GET http://label-studio/api/tasks/xxx
           ▼
    （从 Label Studio 拉原始数据文件/任务数据）
```

### 关键设计哲学

1. **HTTP 无状态协议** —— 不用 RPC、不用 gRPC、不用 Kafka，就最朴素的 HTTP
2. **双向通信** —— Label Studio 能调模型，模型也能回调 Label Studio 拉数据
3. **SDK 是可选的** —— 官方给 Python SDK，但你完全可以用 Node.js、Go、Rust 手写

### 4 个核心端点

#### 1. `POST /setup` —— 握手端点
- **触发时机**：项目首次绑定这个 ML Backend 时
- **干的事**：模型缓存 `label_config`、保存 `hostname + access_token`、报告 `model_version`

#### 2. `POST /predict` —— 最核心的端点
- **批量模式**：tasks 数组，context 为 null
- **交互模式**：tasks + context（用户刚刚做的操作）
- **响应**：results 数组，每项含 model_version、score、result

#### 3. `POST /train` —— 训练端点
- **请求内容**：event（如 ANNOTATION_CREATED）+ annotations 数组
- **响应**：新的 model_version
- **注意**：训练通常是**长任务**，Label Studio 不等完成，请求立即返回 `202 Accepted`

#### 4. `POST /webhook` —— 事件订阅端点
Label Studio 会往这里推各种事件：ANNOTATION_CREATED、ANNOTATION_UPDATED、PROJECT_UPDATED 等。
模型可以在这里实现**主动学习触发**。

### 20 个必须注意的细节

#### 🎯 协议层面
1. **`from_name` / `to_name` 必须动态从 label_config 里读，别硬编码**
2. **`type` 字段必须和标注工具类型严格一致**（大小写敏感）
3. **`original_width` / `original_height` 不能省**
4. **百分比坐标范围是 0-100，不是 0-1**
5. **`score` 字段影响多处**（置信度排序、主动学习、Data Manager 筛选）

#### 🌐 网络 / 部署层面
6. **`LABEL_STUDIO_URL` 里写 `localhost` 是最常见的坑**
7. **大批量预测会超时**（建议按 task 粒度异步调用）
8. **GPU 并发控制**（用信号量或请求队列）
9. **健康检查端点要轻量**（只返回版本号和状态）

#### 📦 数据访问层面
10. **`get_local_path()` 处理了三种 URL 形态**（直连、LS 内部、云存储）
11. **云存储文件缓存默认是永久的**

#### 🔐 安全层面
12. **API Token 泄漏风险**（用项目级 Token 或只读 Token）
13. **Basic Auth 不够用时支持 Token**

#### 🔄 交互式模式的陷阱
14. **`context` 可能是 None 也可能是空对象**
15. **交互模式必须快（<1s）**
16. **交互模式返回的预测应该"覆盖"而非"追加"**

#### 🔁 训练闭环的陷阱
17. **不要同步等训练完成**（用异步线程）
18. **训练期间 `/predict` 要用旧权重**（维护两份模型）
19. **`model_version` 变更后前端能看出差异**

#### 🧪 容错与可观测
20. **单条失败不要让整批预测崩溃**

### 插座设计背后的 5 个关键原则

#### 原则 1：**协议大于 SDK**
协议（HTTP + JSON Schema）是合同，SDK 只是参考实现。用户可以完全不用 Python，用任何语言实现这个协议都能工作。

#### 原则 2：**无状态 + 回调**
Label Studio 调模型时不塞完整数据（图片/视频太大），只给 URL。模型需要数据时反向调 Label Studio 拿。

#### 原则 3：**类型擦除（Type Erasure）**
不管模型是分类还是检测还是分割，返回的都是同一个 JSON 形状。前端用 `type` 字段做 discriminator。

#### 原则 4：**预测与标注数据双轨**
Prediction 永远不会覆盖 Annotation，两者并存在数据库里。这让事后复盘成为可能，也是主动学习的基础。

#### 原则 5：**配置驱动，不是代码驱动**
label_config (XML) 是声明式的，下发给模型后模型再按此适配自己的输出。这让**同一个模型服务多个标注任务**成为可能。

## 八、四档模型兼容的设计哲学

### 核心洞察：兼容性来自"抽象层够高"

Label Studio 没有为不同模型写不同的兼容代码，它做对了一件事：**让协议本身对"模型类型"一无所知**。

兼容性不是"写了四套适配逻辑"，而是来自 5 个**精心去耦合的设计决策**。

### 5 个让协议变通用的设计决策

#### 决策 1：**类型擦除的输出**
所有模型的输出都套进同一个"信封"，用 `type` 字段做鉴别器。前端根本不关心这个预测是谁生成的——它只看 `type` 查表决定怎么渲染。

#### 决策 2：**`label_config` 作为"双向声明"**
XML 配置是兼容能力的核心。它扮演**两个相反方向的角色**：

- 告诉前端怎么渲染
- 告诉模型该输出什么结构

**不同类型的模型对这份 XML 有不同的用法**：

| 模型档位 | 怎么用 label_config |
|---|---|
| 封闭集 BERT | 在 `setup()` 时解析，建立 `{native_label → xml_label}` 映射表 |
| 开放集 LLM | 在 `predict()` 时解析，把标签抽出来塞进 Prompt 模板 |
| 类别无关 SAM | 只看工具标签定位，不关心标签内容 |
| 少样本模型 | 解析标签集合，把任务第一批标注作为 few-shot 示例 |

#### 决策 3：**`context` 作为多态输入**
这是让批量 + 交互式统一在同一个 `/predict` 端点里的关键设计。

不同的模型对 context 有不同的响应策略：
- 封闭集：通常忽略
- 开放集（GPT）：把 context 里的交互转成"继续对话"
- SAM：**核心依赖**，context.result 就是用户点击的点/框
- 少样本：用 context 扩充支撑集

#### 决策 4：**`extra_params` —— 兼容异常的最后退路**
协议不可能预见所有模型的需求，所以留了一个口子。这些参数在 setup 时下发给模型，模型想怎么用都行。Label Studio **完全不理解这些参数的含义**，只负责透传。

#### 决策 5：**`model_version` 自报 + 自适应训练**
训练这件事在四档模型里差异最大。Label Studio 的处理方式是**完全不干涉**——只做两件事：
1. 把标注数据塞给模型
2. 接收模型回报的新 `model_version`

至于模型内部发生了什么——它根本不关心。

### 逐档模型走一遍

| 阶段 | 封闭集 (BERT) | 开放集 (GPT) | 类别无关 (SAM) | 少样本 |
|---|---|---|---|---|
| **`/setup`** | 解析 XML 建立标签映射表 | 解析 XML 抽出标签列表存起来 | 只记录位置，加载权重 | 解析 XML，初始化空原型集 |
| **`/predict`** (batch) | 原生输出 → 查表翻译 | 构造 Prompt 调 GPT 解析文本 | ❌ 不支持 | k-NN 返回最近类 |
| **`/predict`** (interactive) | ❌ 一般不支持 | 把 context 转成"用户追问" | ✅ **核心**：读 context.result 作为点提示 | 用 context 作为反馈 |
| **`/train`** | 积累 annotations，SGD 微调 | 把 corrections 转成 few-shot，更新 Prompt | ❌ 不支持 | 新 annotation 加入原型集 |
| **`model_version`** | `"bert-v1.2"` → `"bert-v1.3"` | `"gpt4-promptV3"` → `"gpt4-promptV4"` | 永远是 `"sam-vit-h-1.0"` | `"proto-n50"` → `"proto-n120"` |

**协议没变，实现方式完全不同**——这就是兼容的真相。

### 设计代价

世界上没有免费的午餐。Label Studio 为了通用性，牺牲了以下几点：

1. **运行时类型错误** —— `type` 字段是字符串，协议层不能在编译期保证模型输出形状和 type 匹配
2. **适配代码沉在 userland** —— 模型接入者必须自己写适配逻辑
3. **前端渲染复杂度高** —— 要维护 20+ 种 type 的渲染器
4. **调试困难** —— 协议不强校验，错了只会静默失败

### 核心结论

> Label Studio 的"四档兼容"不是写了四套代码，而是做了一件更漂亮的事——**让协议对"模型类型"完全透明**。
> 
> 5 个关键设计：
> 1. **类型擦除** —— type 字段做 discriminator，输出统一信封
> 2. **label_config 双向声明** —— 协议不假设谁先知道标签
> 3. **context 多态输入** —— 批量和交互共用一个端点
> 4. **extra_params 退路** —— 不可预见的需求有地方塞
> 5. **model_version 自报** —— "训练"的语义由模型自己定义

## 九、开放集模型（LLM）预标注完整链路

### 场景设定

**任务**：对 5 条金融新闻做情感标注  
**标签 schema**：正面 / 负面 / 中性  
**模型**：`llm_interactive` backend + GPT-4  
**用户动作**：Data Manager 选中 5 条任务 → Retrieve Predictions

### 阶段 0：Setup（一次性握手）

项目刚绑定 backend 时，Label Studio 发一次 `POST /setup`，包含：
- project id
- label_config（XML）
- hostname + access_token
- extra_params（OPENAI_API_KEY、PROMPT 模板、TEMPERATURE、MODEL_NAME）

**模型侧干两件事**：
1. 存档 label_config（后面 predict 时要解析它）
2. 从 extra_params 读取用户自定义的 Prompt 模板 + API Key

此刻 **label_config 被模型记住了**，但还没开始用——它会在每次 predict 时再拿出来解析。

### 阶段 1：用户触发预标注

Data Manager 里选 5 条任务 → Actions → Retrieve Predictions。Label Studio 后端把这 5 条打包成一次请求发到模型的 /predict 端点。

**关键观察**：
1. `label_config` 在请求里**又发了一次**（保证幂等）
2. `context` 为 null，表示这是批量模式
3. `tasks` 是数组，模型得一次处理 5 条

### 阶段 2：模型内部的 label_config 解析

这是开放集模型最关键的一步——**从 XML 里抽出"当前任务的标签列表"**：

```
从 parsed_label_config 里可以拿到：
- control_name: "sentiment"
- to_name: "text"
- input_field: "text"（任务数据里的字段名）
- labels: ["正面","负面","中性"]  ← 核心！
```

**这一步是封闭集和开放集的分水岭**：
- 封闭集模型（BERT）在 setup 就把标签映射表建好了
- 开放集模型**每次 predict 都重新读 label_config**，因为标签是"调用时"才决定的

### 阶段 3：Prompt 构造 —— 开放集的核心魔法 ✨

这里是整个流程最精彩的地方。模型要把一份 XML 配置**翻译**成一段 LLM 能理解的自然语言指令。

#### 3.1 基础 Prompt 模板（简单版）

```
分类下面这段金融文本的情感倾向

可选标签：正面, 负面, 中性

文本：央行今日公布最新货币政策，维持基准利率不变。

请只返回一个标签名，不要任何解释。
```

#### 3.2 带 Schema 约束的 Prompt（生产级做法）

上面的简单 Prompt 写法 LLM 可能返回"我认为是中性"之类的啰嗦回答。**生产环境必须用"Structured Output"或"Function Calling"强约束**：

**方式 A：OpenAI Function Calling（最可靠）**

把"标签列表"变成函数签名的 `enum` 参数。这样 GPT 被强制返回结构化数据，且输出一定落在标签列表里。

**方式 B：OpenAI Structured Outputs（2024 后的更干净做法）**

动态构建 Pydantic 模型，用 Literal 类型枚举。

**方式 C：JSON Mode + 失败重试（兼容老模型 / 开源 LLM）**

让 LLM 返回严格 JSON，模型侧做格式校验 + 重试 + 降级。

#### 3.3 Few-shot 示例注入

为了提升精度，一般还会在 Prompt 里塞 2-3 个示例：

```
分类下面这段金融文本的情感倾向

可选标签：正面, 负面, 中性

参考示例：
- 文本："公司Q3净利润同比增长85%"
  标签：正面
- 文本："监管层约谈相关企业"
  标签：负面
- 文本："市场开盘平稳"
  标签：中性

待分类文本：央行今日公布最新货币政策，维持基准利率不变。
```

**Few-shot 数据从哪来？**
- 从 Label Studio 回拉当前项目的前 N 条已标注任务
- 通过 `hostname + access_token` 调 `/api/tasks?project=42&completed=true&limit=3`
- 把标注结果转成 "文本 → 标签" 对作为 few-shot

这就是**"开放集模型也能随着标注积累越来越准"**的秘密——虽然没有真训练，但 few-shot 示例变多了。

### 阶段 4：批量调用 LLM 并聚合结果

模型侧的处理流程：
1. 从 label_config 抽出标签
2. 拉取 few-shot 示例
3. 逐条调用 LLM（可并发优化）
4. 按协议格式封装

**并发优化（生产必做）**：LLM 每次 API 调用 1-3 秒，5 条串行要 15 秒，必须并发。但要注意 OpenAI API 的 RPM/TPM 限流，生产环境通常还要加指数退避重试。

### 阶段 5：模型返回响应

模型返回符合 LS 协议的 JSON，每个 task 一条结果，包含：
- model_version
- score（任务级置信度）
- result 数组，每项含 from_name、to_name、type、value、score

### 阶段 6：Label Studio 入库与 UI 渲染

Label Studio 后端收到响应后：
1. 按任务把预测数据分别存入 Prediction 表
2. Prediction 表独立于 Annotation 表（双轨设计）
3. 前端按 type 字段分派渲染器

标注员打开任务时看到的效果：带"模型预测"徽章的选项（置信度 94%）+ 接受/拒绝按钮。

### 阶段 7：交互式模式的差异（Smart Tool）

如果用户在 UI 勾选了 "Interactive preannotations"，流程变成"**标注员操作 → 实时 LLM 追问**"：

比如对于 LLM 任务，标注员在文本某段上划选高亮，前端立刻发出带 context 的请求。模型侧：
- 如果 context 有内容：基于用户高亮的片段做二次分析
- 否则走批量分支

**交互式模式本质是把 `context` 转成"用户在追问"**——LLM 天生擅长对话，所以这种改造很自然。

### 关键细节汇总：开放集的 8 个必须注意

| # | 细节 | 踩坑后果 |
|---|---|---|
| 1 | **标签列表每次从 label_config 动态读** | 用户改了标签 UI，模型还在用旧标签 → 预测对不上 |
| 2 | **必须用 Structured Output / Function Calling 强约束** | LLM 返回非法值 |
| 3 | **Schema 里用 `enum`/`Literal` 把标签钉死** | 10% 的调用会幻觉出新标签 |
| 4 | **置信度怎么算** | LLM 本身不返回概率，只能让它自报或用 logprobs 反推 |
| 5 | **Prompt 模板要支持 i18n** | 英文 Prompt 遇到中文标签/文本可能答非所问 |
| 6 | **Few-shot 示例要覆盖所有标签** | 只给"正面"示例，模型会偏置 |
| 7 | **TPM/RPM 限流 + 指数退避** | 并发 100 条直接 429 |
| 8 | **Token 成本监控** | 标注 10 万条可能几千美元 |

### 置信度的三种获取方式

- **方式 1：让 LLM 自报**（最简单但不可靠）
- **方式 2：用 logprobs 反推**（准确但需要模型支持）
- **方式 3：多次采样投票**（稳但贵）

### 核心结论

> **开放集模型的预标注链路本质是"一次 XML → 一段 Prompt → 一次 LLM 调用 → 一份协议格式"的四段式翻译**。
> 
> 其中最精彩的魔法发生在**标签从 XML 变成 Prompt** 的那一步——通过 `parsed_label_config["labels"]` 抽出标签列表 + Function Calling 的 `enum` 约束，把"开放集模型"**动态收敛到当前任务的封闭集**上，既保留了 LLM 的灵活性，又保证了输出合规性。

**一半以上的代码是"开放集特有"的工作**——这也是为什么开放集模型接入的门槛比封闭集高。

### 🔮 LLM API（外部服务）

这是整个链路里**唯一"买服务"**的部分。职责非常聚焦：

| 职责 | 谁提供 |
|---|---|
| **模型推理本身** | OpenAI / Anthropic / Azure / 自建 vLLM |
| **Function Calling / Tool Use 约束** | 现代 LLM API 原生支持 |
| **Structured Output 保证** | OpenAI `response_format=Pydantic` 等 |
| **Logprobs 输出**（做置信度用） | 部分 API 支持 |
| **并发与 SLA** | API 供应商的责任 |

**注意**：LLM API **对 Label Studio 一无所知**——它只知道自己收到了一段 Prompt、要按 Schema 返回 JSON。这保证了 LLM 可以随便换（GPT-4 → Claude → Qwen），不影响平台任何代码。

| 阶段 | 🟦 Label Studio | ⚙ ML Backend | 🔮 LLM API |

| **4. 拉 few-shot 示例** | 响应 `GET /api/annotations` | 发起 HTTP GET，拿到历史标注；转成 "input → label" 对 | — |
| **5. 构造 Prompt + Schema** | — | 模板渲染；生成 Pydantic Literal | — |

### 责任边界一句话总结

| 边界 | 由谁维护 | 变动影响 |
|---|---|---|
| **用户 UI ↔ Label Studio** | Label Studio | 改 UI 不影响模型 |
| **Label Studio ↔ ML Backend** | **协议固化**（双方都得遵守） | 改协议双方都要改 |
| **ML Backend 内部** | 模型作者 | 改逻辑不影响 LS |
| **ML Backend ↔ LLM API** | 模型作者选型 | 换 LLM 只影响 Backend |

**这三层解耦是 Label Studio 架构最值钱的地方**——它让"平台厂商"、"模型服务方"、"LLM 供应商"三个角色可以独立演化、各司其职。

### 核心结论

> **预标注链路在三个主体间做了非常漂亮的职责切分**：
> - 🟦 **Label Studio 只关心"怎么把任务送进来、把预测显出去"**（交互层）
> - ⚙ **ML Backend 只关心"怎么把任务翻译成模型能懂的样子，再翻译回来"**（编排层）
> - 🔮 **LLM API 只关心"给定输入，按 Schema 输出"**（算力层）
> 
> 中间靠两道 HTTP 契约连接：
> - 🟦↔️⚙ 是 **Label Studio 定义的协议**（4 端点 + from_name/to_name/type 三元组）
> - ⚙↔️🔮 是 **LLM 厂商定义的 API**（OpenAI Chat Completion 等）
> 
> 这种切分让三方可以完全并行演化，也是 Label Studio 能做成开源标注平台"事实标准"的架构根基。

## 附录：术语速查

| 术语 | 含义 |
|---|---|
| **预标注 (Pre-annotation)** | 模型对待标注数据先给出候选结果，供标注员审校 |
| **批量预标注 (Batch)** | 任务级别的预标注，整批数据一次性跑完 |
| **交互式预标注 (Interactive)** | 操作级别的预标注，标注员每次操作都触发模型 |
| **Smart Tools** | 带 `smart="true"` 的工具，会请求 ML Backend 实时辅助 |
| **Prediction** | 模型预测结果，独立于 Annotation 存储 |
| **Annotation** | 人工标注最终结果 |
| **label_config** | XML 声明的标注工具配置，同时是 UI 渲染契约和模型输出契约 |
| **from_name / to_name** | 预测结果绑定到标注工具实例的键 |
| **type** | 预测结果的类型鉴别器（choices/rectanglelabels/brushlabels 等） |
| **context** | 交互式模式下，用户操作的描述 |
| **extra_params** | 模型接入时的自定义配置槽 |
| **model_version** | 模型自报的版本号，用于 A/B 对比 |
| **主动学习 (Active Learning)** | 模型对未标注样本打不确定性分数，优先分配难样本给人工 |
| **四档模型** | 封闭集 / 可配置集 / 开放集 / 少样本（按标签灵活度分类） |
| **Prompt 工程** | 开放集模型的核心配置方式，通过 Prompt 模板引导 LLM 输出 |
| **Function Calling** | LLM API 的结构化输出约束机制，用 JSON Schema 规范输出 |
