## 1. 标注团队（Annotation Team）

### 1.1 背景与目标

平台原有"组织 / 成员 / 角色"三层权限模型不能很好地表达"**一批标注员作为一个独立可复用实体被多个空间使用**"这一业务诉求。为此引入"标注团队"概念：

- **以标注团队为边界**管理标注团队（同一标注团队内的成员可在空间管理的任意空间中复用）。
- 每个团队内的成员角色需在原本基础上额外为之赋予**标注员**角色。
- 支持通过**按账号跨组织精确查找**的方式拉入其它组织的标注员（体现平台侧对跨组织协作的一级支持）。

> **路由与菜单位置**：`/console/annotation-teams`，控制台 → **权限管控** 菜单组下，位于「成员管理」与「角色管理」之间。
> 对应实现文件：`src/pages/console/ConsoleAnnotationTeams.tsx`、`ConsoleAnnotationTeamDetail.tsx`。

### 1.2 功能架构

```text
标注团队列表页
├── 顶部 3 张统计卡片（团队总数 / 总标注成员 / 任务总量）
├── 搜索栏（团队名称、团队标识）
├── 团队表格（支持任务总量排序）
│   ├── 点击团队名称 → 团队详情页
│   └── 行操作：查看详情 / 编辑 / 删除（带确认弹窗）
├── 「新增标注团队」按钮 → 新增弹窗
│
团队详情页
├── 顶部：团队名 / 状态徽章 / 标识 / 所属组织 / 负责人
├── Tab 1：成员管理（本次范围）
└── Tab 2：团队绩效（占位中，"敬请期待"）
```

### 1.3 功能需求

#### 1.3.1 列表页统计卡片

| 指标 | 定义 |
| --- | --- |
| 团队总数 | 当前可见的所有标注团队条数 |
| 总标注成员 | 所有团队的成员账号累加（不同团队的重复账号需**按团队各计一次**，不做去重；因为绩效、工单均按团队计算） |
| 任务总量 | 所有团队 `taskTotal` 字段之和 |

#### 1.3.2 列表字段与交互

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| 团队名称 | 文本 | 支持按名称模糊搜索；点击跳详情 |
| 团队标识 | 英文/下划线 | 系统以名称自动生成，唯一约束 |
| 所属组织 | 文本 | 来自创建时选择的关联组织 |
| 负责人 | 文本 | 来自创建时选择的组织成员 |
| 成员数 | 数值 | `TEAM_MEMBERS[id].length` |
| **任务总量** | 数值 | **支持点击表头排序**（desc ↔ asc） |
| 描述 | 文本 | 最长 300 字，行内 truncate + title 显示 |
| 创建时间 | 日期 | |
| 操作 | 图标 | 查看详情 / 编辑 / 删除（带二次确认） |

#### 1.3.3 新增标注团队

弹窗字段（所有必填项标红星）：

| 字段 | 必填 | 约束 |
| --- | :-: | --- |
| 团队名称 | ✅ | 中英文 / 数字 / 下划线，长度 **2-30**；重名校验 |
| 关联组织 | ✅ | 下拉选择，候选为当前用户有权限的组织 |
| 负责人 | ✅ | 下拉选择，候选为所选组织的成员（带账号显示）|
| 描述 | ❌ | 0–300 字符，实时字数统计，超 280 字变红预警 |

- 未通过校验时，`创建` 按钮禁用。
- 创建成功：Toast 提示 "标注团队创建成功"，关闭弹窗，列表刷新。

#### 1.3.4 团队成员管理（详情页 Tab 1）

**表格字段（均支持点击表头排序）：**

| 字段 | 说明 |
| --- | --- |
| 姓名 / 账号 | 基础信息 |
| **任务总量** | 该成员承接的标注任务数量 |
| **标注总量** | 已完成标注样本数 |
| **有效标注时长 (h)** | 保留 1 位小数 |
| 加入时间 | 加入本团队的日期 |
| 状态 | `启用 / 停用` 徽章 |
| 操作 | 查看绩效（跳转 `/data-annotation/performance`）/ 移除 |

**添加成员弹窗**（两种来源、统一确认）：

1. **从组织成员中选择**
   - 候选集：**仅限 `MOCK_ORG_ANNOTATORS` 中 `roles` 包含"标注员"**的成员，且排除已在团队的人。
   - 支持姓名 / 账号 / 部门模糊搜索。
   - 勾选后进入"已选成员"区。

2. **按账号精确查找（跨组织）**
   - 输入完整账号 → 点击"查找"。
   - 命中：显示该用户 + "跨组织"徽章 → 点"添加"进入"已选成员"。
   - 未命中 / 已在团队 / 已在待添加列表：Toast 报错。

3. **已选成员**区域可逐个移除；底部统一 "确认添加 (N)" 提交。角色默认且固定为"标注员"，UI 明确提示"不可修改"。

**移除成员**：二次确认弹窗。

#### 1.3.5 团队绩效（详情页 Tab 2）

本期占位 —— 展示 "该模块的功能待后续完善，敬请期待"。后续迭代将对接我的绩效页面。

### 1.4 非功能与校验要求

- 所有按钮在操作进行中需 loading 或 disabled 防重复提交。
- 删除团队 / 移除成员必须二次确认。
- 团队下无法修改成员角色（角色恒为标注员），减少运维混淆。
- 列表分页按前端 mock 处理即可；后端上线后按"共 X 条、20 条/页、最多 1000 页"设计。

### 1.5 验收标准

- [ ] 菜单项出现在 控制台 → 权限管控 → 成员管理 与 角色管理 之间。
- [ ] 可以创建、编辑、删除标注团队，并满足字段级校验。
- [ ] 团队详情页可以添加组织内标注员、跨组织按账号拉入、移除成员，且角色恒为标注员。
- [ ] 任务总量列支持在列表页和详情页做排序切换。
- [ ] 团队绩效 Tab 展示占位页。

## 2. 空间管理 · 权限管理功能

### 2.1 背景与目标

> **入口**：控制台 → 空间管理 → 点击任意空间 → **权限管理** Tab。
> 对应实现文件：`src/pages/console/ConsoleSpaceDetail.tsx` 中的 `PermissionsTab`。

空间需要区分两种本质不同的成员入口，以**最小化重复劳动**并保证"标注团队"这一一级实体的复用：

1. **核心成员**：通过手动挑选 / 账号查找的方式加入，可担任任意一个或多个空间角色（如空间管理员、数据工程师、质检员等）。
2. **标注团队成员**：通过关联"标注团队"整包拉入。成员角色固定为 "标注员（团队名称）"，不可单独编辑 / 移除，移除需解绑整个团队。

同一账号如果**同时**是核心成员与团队成员：以"核心成员"身份保留在列表中，但**额外带上所关联团队的归属标记**（见 2.3.6 合并逻辑）。

### 2.2 空间角色集合

固定 7 个空间级角色（`SPACE_ROLES`）：

```text
空间管理员 / 数据工程师 / 标注员 / 质检员 / 验收员 / 仲裁员 / 观察员
```

每个角色对应一组空间内数据/任务层面的权限集合（本 PRD 不展开 CRUD 矩阵，交由后端权限清单文档）。一个核心成员可持有**多个**角色；团队成员恒为"标注员（团队名）"一个角色。

### 2.3 功能需求

#### 2.3.1 筛选与总览

- **搜索框**：按姓名 / 账号模糊搜索。
- **角色筛选**：下拉，选项 = `"全部角色" + SPACE_ROLES`。特殊匹配规则：
  - 选择"标注员"时，核心成员角色含"标注员"的 **或** 任何带 `标注员（…团队名…）` 标签的团队成员都要命中。
  - 选择具体 `"标注员（xx团队）"` 时，只命中对应团队的成员。
- **角色统计徽章区**：列出当前空间中存在的角色及其人数（核心成员按 `SPACE_ROLES` 拆分；团队成员按 `"标注员（团队名）"` 分组）。

#### 2.3.2 成员表格

| 字段 | 说明 |
| --- | --- |
| 姓名 / 账号 | 基础信息 |
| 角色 | **核心成员**：多个 `primary` 徽章（紫色系）；**团队成员**：单个 `indigo` 徽章 `标注员（xx 团队）` |
| 加入时间 | 加入本空间的日期 |
| 状态 | 启用 / 停用 |
| 操作 | 编辑角色 / 移除；**团队成员这两个图标均 disabled**，hover 气泡提示"标注团队成员不可单独编辑 / 移除" |

排序规则：**核心成员优先**（保持原有顺序），**团队成员置底并按团队名字典序分组**。

#### 2.3.3 新增成员入口（按钮下拉）

点击"新增成员"按钮（或 hover）展开小菜单：

1. **新增核心成员** → 打开"新增核心成员"弹窗（见 2.3.4）
2. **添加标注团队** → 打开"添加标注团队"弹窗（见 2.3.5）

> 设计意图：两种入口使用不同的实体模型，**不混用**。

#### 2.3.4 新增核心成员 弹窗

三段式结构：

1. **从组织成员中选择**
   - 候选：`MOCK_ORG_MEMBERS` 中尚未加入当前空间、也未在"已选"列表中的成员。
   - 支持姓名 / 账号 / 部门搜索。
   - 勾选后进入"已选"。

2. **按账号精确查找（跨组织）**
   - 与标注团队一致的跨组织拉人机制。

3. **已选成员** 区
   - 每个已选成员**独立配置多个**空间角色，默认勾选第二个角色（`数据工程师`）。
   - 至少保留 1 个角色；若用户取消全部勾选，保持原状不变（UI 防御）。
   - 可逐个移除。

底部 "确认添加 (N)" 一次性入列。

#### 2.3.5 添加标注团队 弹窗

- 展示同组织内的所有标注团队的**勾选表格**（表头：团队名 / 所属组织 / 描述 / 负责人 / 成员数）。
- 已关联的团队默认勾选；取消勾选代表解绑；新勾选代表关联。
- "确认 (N 个团队)" 提交；按差集：
  - **新增的团队**：把 `TEAM_MEMBERS[team.id]` 展开为成员逐个加入空间成员列表，带上 `linkedTeamId / linkedTeamName`。
  - **取消的团队**：移除所有 `linkedTeamId` 匹配的成员（含并列核心成员的，见 2.3.6）。
- Toast 分别反馈关联 / 移除数。

#### 2.3.6 核心成员 与 团队成员 的合并逻辑

| 现状 | 新增关联该团队 | 结果 |
| --- | --- | --- |
| 账号不在空间 | ✅ 新增为团队成员 | 一条成员记录，`linkedTeamId = 团队` |
| 账号已是核心成员 | ✅ 给该核心成员打上 `linkedTeamId/Name` 标记 | 一条成员记录，角色= 核心成员角色，但"移除"受团队规则影响？见下条 |

> **规则**：当核心成员也属于某团队时，列表里**合并展示**为核心成员（角色保留），但会在角色列同时体现其团队归属。暂不允许在 UI 上仅移除"团队来源"那一部分——要么用"移除核心成员"将其完全踢出（会破坏团队关系 → 解绑团队时仍会清掉），要么"取消关联团队"清掉该账号的团队归属。

#### 2.3.7 编辑角色 弹窗（仅核心成员）

- 弹窗内 checkbox 多选 `SPACE_ROLES`。
- 至少 1 个角色，否则最后一个勾选项禁止被取消。
- 确认后成员的 `roles` 字段被替换为新集合。

#### 2.3.8 移除成员（仅核心成员）

- 二次确认弹窗，提示"移除后该成员将无法访问本空间的资源"。
- 确认后从成员列表移除；团队成员的移除必须**通过解绑团队**完成。

### 2.4 非功能要求

- 弹窗内组织成员列表与已选列表需在新增 / 取消时同步 —— 同一账号不可同时出现在两处。
- 跨组织查找不做网络请求，当前以 mock 直接造出 `用户_{account}` 对象。后端联调时替换为用户中心搜索接口。
- 权限变更操作均应有 Toast 反馈与 undo 提示（后端上线后）。

### 2.5 验收标准

- [ ] 权限管理 Tab 能分辨 "核心成员" 与 "团队成员" 两类条目。
- [ ] 新增核心成员支持跨组织查找、为每个候选独立配置多角色。
- [ ] 添加标注团队勾选 / 取消勾选后，成员列表中团队成员按团队分组置底。
- [ ] 团队成员的"编辑角色 / 移除"按钮 disabled。
- [ ] 角色筛选器能正确识别 `"标注员"` 与 `"标注员（xx 团队）"`。
- [ ] 角色统计徽章覆盖所有现存角色与团队。

## 3. 创建文本 SFT / RLHF / DPO / KTO 数据集 业务流程

### 3.1 背景与目标

平台需要支持大模型训练主流范式的数据集：**SFT（监督微调）、RLHF（人类反馈强化学习）、DPO（直接偏好优化）、KTO（Kahneman-Tversky Optimization）**。这些数据集与"通用文本"最大的差异是：

- **文件格式严格**：仅允许 JSONL。
- **每行字段固定**：不同范式字段结构不同。
- **数据质量直接影响微调效果**，因此在创建、导入、查看全链路需要为用户提供"结构化引导"与"示例下载"。

> 实现入口：
> - 入口页：`DataManagementDatasets.tsx`
> - 基本信息与权限目录：`DatasetCreateForm.tsx`
> - 版本与导入：`DatasetVersionList.tsx` → `DatasetVersionCreate.tsx` → `DatasetImportConfig.tsx`
> - 导入后的版本详情：`DatasetVersionDetail.tsx`

### 3.2 业务流程总览

```text
[数据集管理列表]
    │ 点击「新增数据集」
    ▼
[创建数据集表单]
    · 填写基本信息（模态=文本 + 大模型数据集类型=SFT/RLHF/DPO/KTO）
    · 填写版本信息（V1.0、存储位置、存储格式、版本描述）
    · 配置权限（授权范围、用户/角色、权限点）
    · 配置数据目录（系统目录 + 自定义目录）
    │ 提交 → 创建成功
    ▼
[版本列表]
    · V1.0 新版本，状态 "创建中"
    │ 点击「导入数据」
    ▼
[导入数据配置页]
    · 数据集模态、标注状态、上传方式 = 本地导入
    · 格式选择 = 仅可选 "JSON 文件"（强约束）
    · 文件上传区右上角 ← 「下载 文本 SFT/RLHF/DPO/KTO JSONL 示例文件」（蓝色链接）
    · 上传若干 .jsonl 文件
    │ 开始上传
    ▼
[版本详情页]
    · 实时进度：「[时间戳] 共 X 个文件需要导入，已完成 Y，失败 Z」
    · 支持暂停 / 续传
    · 成功后版本标记 "可用"
```

### 3.3 功能需求

#### 3.3.1 创建数据集 · 基本信息（Section 1）

| 字段 | 必填 | 规则 |
| --- | :-: | --- |
| 数据集中文名称 | ✅ | 中英文 / 数字 / 下划线，长度 2–50 |
| 数据用途 | ✅ | 单选 `模型微调 / 预训练 / 其他`（本范式默认选 "模型微调"） |
| 数据集模态 | ✅ | 下拉 `文本` |
| **大模型数据集类型** | ✅ | **仅当模态=文本时出现**，下拉 `通用文本 / 文本 SFT / 文本 RLHF / 文本 DPO / 文本 KTO` |
| 数据集描述 | ❌ | 0–1024 字符，右下角计数 |
| 数据集标签 | ❌ | 最多 20 个 K-V 标签；独立的标签编辑弹窗 |

#### 3.3.2 创建数据集 · 版本信息（Section 2）

- 默认创建 V1.0 且不可修改。
- 存储位置（下拉，`STORAGE_LOCATIONS`）。
- 存储格式：`Lance / Parquet / CSV / JSON`，对于本范式**建议使用 Lance 或 Parquet**，CSV/JSON 作为过渡选择。
- 版本描述 0–1024 字符。
- 版本标签（K-V，最多 20 个）。

#### 3.3.3 创建数据集 · 权限配置（Section 3）

- 授权范围：`数据集所有者 / 空间内全体成员 / 指定用户 / 指定空间角色`。
- 当选择 "指定用户" → 出现用户多选；"指定空间角色" → 出现角色多选。
- 当授权范围 ≠ 所有者时，必选一个或多个权限点：`读数据集 / 写数据集 / 创建数据集版本`。

#### 3.3.4 创建数据集 · 目录配置（Section 4）

- 系统目录（必填）：勾选到对应的一级系统目录。
- 自定义目录（选填）。

#### 3.3.5 版本导入（`DatasetImportConfig`）

##### 3.3.5.1 关键约束

```ts
const isLargeModelText = dataset.modality === "文本" &&
  ["文本 SFT", "文本 RLHF", "文本 DPO", "文本 KTO"].includes(dataset.type || "");
```

当 `isLargeModelText === true`：

1. **格式卡片仅保留 `"JSON 文件"` 一项**（其他格式被过滤，UI 不展示）。
2. 默认 `fileFormat = "JSON 文件"`，用户无需选择。
3. 在**文件上传 Section 的标题右侧**渲染一条蓝色字体的下载链接：
   - 文字：`下载 <类型> JSONL 示例文件`（如"下载 文本 SFT JSONL 示例文件"）
   - 图标：`lucide-react` 的 `Download`（`w-3.5 h-3.5`）
   - 样式：`text-blue-600 hover:text-blue-700 hover:underline`
   - 行为：点击后通过 `Blob + URL.createObjectURL` 在浏览器端下载一份 `.jsonl` 文件；文件命名为 `sample_<类型英文化>_data.jsonl`（如 `sample_文本_sft_data.jsonl`），内容见 3.4。

> **注**：示例 JSONL 的生成逻辑 `getJsonlSampleContent(textType)` 同时支持「通用文本」类型的样本内容，当用户通过其他入口（例如后续扩展）选择通用文本时可以直接复用同一套生成器。本期 UI 仅在"文件上传"Section 右上角对 SFT/RLHF/DPO/KTO 四种类型暴露下载入口。

##### 3.3.5.2 其他上传方式

- **平台已有数据集**：从"我的数据集 / 我订阅的 / 分享给我的"选择目标 + 版本 + 导入范围。本范式下**不限制**文件格式（假设源数据已是 JSONL）。
- **在线 FTP 导入**：支持 FTP/SFTP/FTPS，文件类型下拉 `JSON 文件`；支持单次 / 定时同步（Cron 表达式）。测试连接前不允许提交。

##### 3.3.5.3 开始上传

- 点击后若校验通过，Toast 提示 "上传任务已创建，正在处理中..."，跳转版本详情页。
- 上传过程中版本状态 = `导入中`，版本列表不可删除该版本。

#### 3.3.6 版本详情页上的实时进度

- 顶部展示：`模态 / 用途 / 存储 / 范围 / 文件数量 / 版本状态`。
- 当版本仍有未完成或失败的文件时，展示进度条区：

  > 共计 **7** 个文件需要导入，已完成 **5** 个导入，失败 **1** 个 `[2026-04-21 10:45:23]`

  - 时间戳放在文案尾部（灰色、`text-muted-foreground/70`），标识"本次导入批次"发起时间。
- 进度条显示当前批次百分比，提供"暂停 / 续传"。
- 文件层级的状态：`导入中 / 导入完成 / 导入失败 / 解析中 / 解析失败`，失败时展示 `statusMessage` 气泡。

### 3.4 JSONL 字段结构规范与示例

示例文件由前端的 `getJsonlSampleContent(textType)` 生成，`downloadJsonlSample` 调用后在浏览器端触发下载，文件命名为 `sample_<类型英文化>_data.jsonl`（例如 `sample_文本_sft_data.jsonl`、`sample_general_data.jsonl`）。生成器支持以下 **5 种** 类型：`通用文本 / 文本 SFT / 文本 RLHF / 文本 DPO / 文本 KTO`，每种每行一个 JSON 对象（JSONL 规范）。

> **范围说明**：本期 UI 仅对后 4 种（SFT/RLHF/DPO/KTO）在"文件上传"右上角暴露蓝色下载链接；「通用文本」的下载入口目前不在 UI 上直接暴露，仅作为生成器的兜底 default 以及为后续扩展保留。

#### 3.4.1 文本 SFT

采用 OpenAI 风格的 chat messages 结构：

```jsonl
{"messages":[{"role":"system","content":"你是一个有用的AI助手。"},{"role":"user","content":"请介绍一下人工智能。"},{"role":"assistant","content":"人工智能是一门研究如何让计算机模拟人类智能的学科。"}]}
{"messages":[{"role":"user","content":"什么是机器学习?"},{"role":"assistant","content":"机器学习是人工智能的一个分支,它使计算机能够从数据中学习并做出预测。"}]}
```

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | :-: | --- |
| `messages` | Array\<Message\> | ✅ | 多轮对话消息列表，顺序表示交互顺序 |
| `messages[i].role` | `"system"` \| `"user"` \| `"assistant"` | ✅ | 消息角色 |
| `messages[i].content` | string | ✅ | 消息内容文本 |

约束：
- 数组长度 ≥ 1；首条允许是 `system`，但单条只出现一次。
- 最后一条建议为 `assistant` 作为期望输出；`user` 与 `assistant` 交替出现。

#### 3.4.2 文本 RLHF

```jsonl
{"prompt":"请解释量子计算的基本原理。","chosen":"量子计算利用量子力学现象如叠加和纠缠来进行计算,具有超越经典计算机的潜力。","rejected":"量子计算是一种使用量子比特进行运算的计算方式。"}
{"prompt":"如何学习编程?","chosen":"学习编程建议从基础开始,选择一门适合初学者的语言如Python,通过实践项目来巩固知识。","rejected":"你可以随便找本书看看。"}
```

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | :-: | --- |
| `prompt` | string | ✅ | 提示 |
| `chosen` | string | ✅ | 人类偏好的优选回答 |
| `rejected` | string | ✅ | 人类偏好的拒绝回答 |

#### 3.4.3 文本 DPO

字段结构与 RLHF **完全相同**（`prompt / chosen / rejected`）。语义上 `chosen/rejected` 构成偏好对，用于直接偏好优化（DPO）损失。

```jsonl
{"prompt":"写一首关于春天的诗。","chosen":"春风拂面花自开,柳绿桃红映楼台。燕归巢中呢喃语,万物复苏展新颜。","rejected":"春天来了,花开了,树绿了。"}
{"prompt":"推荐一本好书。","chosen":"我推荐《百年孤独》,这是加西亚·马尔克斯的代表作,以魔幻现实主义手法讲述了布恩迪亚家族七代人的故事。","rejected":"看《百年孤独》吧。"}
```

#### 3.4.4 文本 KTO

```jsonl
{"input":"翻译: Hello, how are you?","output":"你好,你怎么样?","label":true}
{"input":"翻译: Good morning!","output":"早上好!","label":true}
{"input":"总结这篇文章。","output":"这篇文章讲了天气。","label":false}
```

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | :-: | --- |
| `input` | string | ✅ | 模型输入 |
| `output` | string | ✅ | 模型输出 |
| `label` | boolean | ✅ | `true` = 期望（desirable），`false` = 不期望（undesirable） |

#### 3.4.5 通用文本（generator 兜底，UI 暂未暴露）

```jsonl
{"text":"这是一条示例文本数据"}
{"text":"这是另一条示例文本数据"}
{"text":"大模型训练需要大量高质量的文本数据"}
```

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | :-: | --- |
| `text` | string | ✅ | 文本内容 |

### 3.5 校验与异常处理

| 场景 | 行为 |
| --- | --- |
| 创建时模态=文本但未选大模型数据集类型 | "请选择大模型数据集类型"（红字），阻止提交 |
| 导入 JSONL 时行解析失败（非法 JSON） | 文件状态 = `解析失败`，`statusMessage` 提示具体行号与原因 |
| SFT：缺失 `messages` 数组或数组元素字段不全 | 文件状态 = `导入失败`，提示"第 N 行缺失字段 `messages` / `role` / `content`" |
| SFT：`messages[i].role` 不属于 `system/user/assistant` | 同上，提示"第 N 行 `role` 取值非法" |
| RLHF / DPO：缺失 `prompt` / `chosen` / `rejected` 任一 | 文件状态 = `导入失败`，提示"第 N 行缺失字段 `chosen`" 等 |
| KTO：缺失 `input` / `output` / `label` 任一 | 文件状态 = `导入失败`，提示"第 N 行缺失字段 `label`" 等 |
| KTO：`label` 字段非 boolean | 同上，提示"第 N 行 `label` 必须为 boolean" |
| 非 UTF-8 编码 | 文件状态 = `导入失败`，提示"编码不支持，请转换为 UTF-8" |
| 单文件超出大小阈值（建议 10GB） | 上传前拦截 |

### 3.6 验收标准

- [ ] 创建页中 `模态=文本` 时出现 `大模型数据集类型` 下拉，5 种类型（含通用文本）可选。
- [ ] 创建后版本 V1.0 自动生成并进入"导入数据"流程。
- [ ] 导入配置页当类型∈{SFT,RLHF,DPO,KTO} 时，`格式选择` 仅展示 `JSON 文件`。
- [ ] "文件上传" Section 标题右上角出现蓝色下载链接，点击可下载对应类型命名的示例 `.jsonl`（`sample_<类型>_data.jsonl`）。
- [ ] 「格式适配配置」卡片内**不再**渲染旧版本的 "XX 格式说明 + 下载示例 JSONL 文件" 辅助卡片（已废弃）。
- [ ] 版本详情页的进度文案末尾含时间戳；文件级失败展示原因。
- [ ] SFT 示例使用 `messages[]` 结构；KTO 示例使用 `input/output/label` 结构；RLHF/DPO 使用 `prompt/chosen/rejected` 结构；均与 3.4 定义严格一致。

## 4. 数据处理 · 工作流调试功能

### 4.1 背景与目标

> 入口：数据处理 → 工作流 → 点击某工作流 → 进入**工作流画布**（`WorkflowCanvas.tsx`，`/data-process/workflow-canvas`）。
> 调试功能与"实例提交运行"是平级的两条提交路径：**调试**用于开发期验证节点间数据链路，**运行**用于正式生产。

### 4.2 画布概念

| 概念 | 说明 |
| --- | --- |
| 节点 | 一个算子实例，有类别、类型、配置、多个输入/输出端口 |
| 节点类别 | `输入节点 / 输出节点 / 文本 / 图像 / 音频 / 视频 / 自定义节点` |
| 连线 | 连接一个节点的输出端口到另一个节点的输入端口 |
| 工作流 | 一组节点 + 连线 + 全局执行配置（并行数、超时、失败策略、重试） |
| 工作流实例 | 以画布快照为基础的一次独立执行记录 |
| 调试模式 | 直接在当前画布上进行一次**非持久化、可采样的**试跑，进入后画布置为只读监控态 |

### 4.3 调试功能的顶层业务流程

```text
编辑画布 ─► 点击 [调试] 按钮
            │
            ├─► 前置校验（validateWorkflow）
            │     ✗ 失败 → 红色弹窗列出错误项，按钮恢复；不进入调试
            │     ✓ 通过
            ▼
[右侧属性面板] → panelMode = "debugConfig"
    · 展示所有输入节点的采样配置（全量 / 比例 / 条数）
    · 展示只读的执行配置摘要（并行数 / 超时 / 失败策略）
    · 蓝色提示条：调试运行将使画布进入只读监控模式
    · 操作：[取消] / [提交调试]
            │ 点击 [提交调试]
            ▼
工作流被自动保存 → startDebug()
    · 构建拓扑顺序（Kahn 算法）
    · 所有节点状态初始化为 pending
    · 从源节点开始依次执行，每节点：pending → running → done/error
    · 失败策略按全局配置：stop / skip / retry（N 次、间隔 S）
    · 调试过程画布只读，禁用 新增节点/连线/移动/删除
            │
            ▼
[右侧监控面板] → rightPanelTab = "debug"
    · 顶部状态徽章：[调试中 / 已完成]
    · 运行时长：实时递增（1s 粒度）
    · 执行进度：完成节点数 / 总节点数 + 进度条
    · 节点执行列表：节点状态、处理数据量、耗时
    · 底部节点日志：点击节点切换日志视图，支持放大 / 关闭
            │
            │ 用户操作：
            ├─► [停止调试] → 终止模拟，清空状态，回到编辑态
            └─► 全部完成 → 徽章变绿，按钮解禁，用户可以修改画布重新调试
```

### 4.4 功能需求

#### 4.4.1 入口按钮与可用性

- 工具栏中的 **调试** 按钮位于 **保存** 与 **运行** 中间。
- 按钮在以下情况**禁用**：
  - 画布为只读 / 运行实例预览态（`isReadOnly` 或 `isExecutionView`）。
- 点击行为：
  - 当前非调试中 → 进入**调试配置**右侧面板。
  - 当前调试中且未完成 → 按钮变红 `停止调试`，点击触发 `stopDebug`。

#### 4.4.2 校验（validateWorkflow）

在进入调试配置前**必须通过**以下校验，否则以弹窗阻塞并高亮有问题的节点：

1. 至少 1 个输入节点 + 1 个输出节点。
2. 每个 `dataset_input` 节点至少配置 1 个变量，且每个变量绑定了数据集；每个 `file_input` 节点必须已上传文件。
3. 每个 `dataset_output` 节点必须填写新数据集名称或选中已有数据集。
4. 所有节点必须位于 Input→Output 的可达路径上（否则报"孤立节点"）。
5. 无环图（DAG）校验。

#### 4.4.3 调试配置面板（`panelMode === "debugConfig"`）

针对**每个输入节点**独立配置本次调试的采样：

| 采样模式 | 额外字段 |
| --- | --- |
| 全量 (`all`) | 无 |
| 按比例采样 (`ratio`) | 1–100 的滑杆 + 右侧百分比展示（默认 10%） |
| 按条数采样 (`count`) | 整数输入（默认 100） |

面板底部展示只读执行配置摘要（并行数 / 超时 / 失败策略），以及一条浅蓝色提示条：

> 调试运行将使用上述参数执行工作流，画布将进入只读监控模式。

操作：
- **取消**：回到 `panelMode = "default"`。
- **提交调试**：触发 `submitDebugRun()`，包含两步：`自动保存当前工作流 + startDebug()`。

#### 4.4.4 调试执行引擎（前端模拟）

- 构建拓扑：使用 Kahn 入度算法构造执行顺序；无入度节点作为起点；未连通节点追加到末尾（**会上报孤立节点 warning**，但不阻止继续执行）。
- 每个节点的模拟：
  - 节点 enter running → 追加两条日志（"开始执行节点..."、"加载数据中..."）
  - 随机延迟 1.2s ~ 2.2s 后完成
  - 完成时随机生成 `count ∈ [500, 10500)` 条、`duration ∈ [2, 32)` 秒
  - 追加日志 "处理 N 条数据" / "节点执行完成 (Ns)"

> 接入后端后，上述模拟应替换为**流式事件推送**（WebSocket / SSE），事件包含 `nodeId / status / progress / log` 字段。

#### 4.4.5 监控面板（右侧）

顶部：
- 标题：`调试监控`（调试）或 `运行监控`（查看实例）
- 状态徽章：`调试中... / 已完成`
- 关闭按钮：折叠右侧面板

中部 · 顶部仪表板：
- 运行时长 `mm:ss` 或 `hh:mm:ss`
- 执行进度：`X / Y 节点` + 水平进度条

中部 · 执行明细：
- 节点卡片列表，每个卡片：
  - 彩色小圆点表示节点类别
  - 节点标签 / 自定义标签
  - 状态图标：等待中（文字）/ 加载动画 / 绿色对勾 / 红色警告
  - 完成后显示 `处理数据量 + 耗时`
- 点击卡片 → 切换下方日志区为该节点日志

底部 · 节点日志：
- 非放大态：固定高度 48 vh
- 放大态（点击 `Maximize2` 按钮）：覆盖画布左侧区域
- 空状态：提示"点击上方节点查看日志"
- 日志每行 `[HH:mm:ss] <消息>` 等宽字体

#### 4.4.6 画布在调试中的只读行为

进入调试后，`isReadOnly` 语义被调试态覆盖：

- 节点：不可拖动、不可删除、不可新增。
- 连线：不可新建 / 删除。
- 节点配置：可查看但不可编辑（未来可放开非关键字段的编辑）。
- 工具栏"保存"按钮：`disabled`（调试完成前）。
- 工具栏"运行"按钮：`disabled`（调试完成前）。
- 允许：右侧面板在 `debug / config` Tab 间切换。

调试完成（`debugAllDone === true`）后，以上按钮全部恢复。

#### 4.4.7 停止与清理

- 用户点击 "停止调试"：
  - 终止定时器 / 清理 setTimeout 回调
  - 清空状态徽章与日志？**保留已产生日志**以便调试排障，但将画布恢复为可编辑态。
  - Toast "调试已停止"
- 用户在调试完成后**编辑画布**（新增节点 / 修改配置）：`debugNodeStates` 会被自动清空，以避免过期状态误导用户。

#### 4.4.8 与"运行实例"的区别（对比）

| 维度 | 调试 | 运行实例 |
| --- | --- | --- |
| 自动保存 | 是 | 是 |
| 生成快照版本 | 否 | **是**（`SNAP-YYYYMMDD-HHmmss`） |
| 写入正式数据集 | 否（临时沙箱 / 采样） | 是 |
| 入参可调整 | **是**（采样） | 否（使用画布当前配置） |
| 持久化日志 | 无 | 有，进入"运行记录" |
| 完成后跳转 | 停留当前画布 | 跳转 `/data-process/run-records` |
| 画布只读 | 调试过程中是 | 提交后是 |

### 4.5 非功能需求

- **性能**：单次调试画布节点数 ≤ 200 时，UI 帧率应 ≥ 45 fps；日志 append 要加节流（后端接入后按 100ms 合批）。
- **一致性**：调试配置在取消后丢弃；调试期间用户切换 Tab / 浏览器最小化不应导致定时器漂移。
- **可观测**：Sentry / 日志平台应记录 `validateWorkflow` 的失败原因、调试启动与停止事件。

### 4.6 验收标准

- [ ] 未通过 `validateWorkflow` 的工作流无法进入调试配置面板。
- [ ] 调试配置面板每个输入节点都能独立设定采样模式并持久化到本次调试。
- [ ] 提交调试后画布进入只读、右侧自动切换到"调试监控" Tab。
- [ ] 节点状态按拓扑顺序 pending → running → done。
- [ ] 节点日志面板支持点击切换、放大、关闭。
- [ ] 全部完成后"保存 / 运行"按钮恢复可用；"调试"按钮可再次进入。
- [ ] 调试中点击"停止调试"应立即停止并保留已产生日志。
- [ ] 调试完成后对画布做任一修改，节点执行状态会自动清空。
